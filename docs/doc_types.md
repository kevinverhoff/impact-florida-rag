# Document Type Reference

Every file in the corpus carries a `doc_type` metadata tag. This tag is the
primary way to scope searches and filters to the kind of evidence you care
about. For example, asking "what do teachers think about coaching?" is very
different depending on whether you want verbatim survey responses
(`feedback_survey_data`) or a synthesized research summary
(`qualitative_theming`).

---

## The 16 Doc Types

### Program & Delivery

| Type | Display label | Description |
|------|---------------|-------------|
| `program_content` | Program Content | Session plans, lesson materials, facilitation guides — the actual curriculum staff deliver |
| `program_overview` | Program Overview | Concept documents, district orientation decks, one-pagers describing what a program is and how it works |
| `program_logistics` | Program Logistics | Operational trackers, participant rosters, application/acceptance records, coaching call logs |

### Evidence & Data Collection

| Type | Display label | Description |
|------|---------------|-------------|
| `intake_survey` | Intake Survey | Baseline assessments and needs/readiness surveys completed before or at program start |
| `feedback_survey_data` | Feedback Survey Data | Raw survey response files where participants evaluate the program — exit tickets, convening feedback, Slido exports, end-of-year surveys. Analysable with `survey_stats`. |
| `teacher_practice_data` | Teacher Practice Data | Raw survey response files where teachers report on their own instructional practice — cycle reflections, site visit surveys, kickoff surveys. Spreadsheets are analysable with `survey_stats`. |
| `other_data_file` | Other Data File | Quantitative data files that don't fit the above — underlying data exports, grade-level visualisations, TW baseline datasets |

### Synthesised Findings

| Type | Display label | Description |
|------|---------------|-------------|
| `qualitative_theming` | Qualitative Theming | Researcher-authored synthesis of structured qualitative data collection — focus group reports, empathy interview write-ups, listening session themes, stakeholder feedback synthesis documents |
| `progress_summary` | Progress Summary | Staff-authored internal monitoring documents — monthly summaries, cadre summaries, learning walk syntheses, historical accounts of a cohort's arc |
| `engagement_notes` | Engagement Notes | Working notes from direct interactions — site visit meeting notes, convening huddle tools, call notes, breakout room notes |

### Reporting & External Artefacts

| Type | Display label | Description |
|------|---------------|-------------|
| `impact_data_report` | Impact Data Report | Polished external-facing data products — impact briefs, annual data slide decks, WestEd/PERTS visualisations, multi-year reports |
| `evaluation_report` | Evaluation Report | Third-party research studies and formal evaluations not authored by IF staff — feasibility studies, usability studies, efficacy registry entries |
| `grants_and_funder_reporting` | Grants & Funder Reporting | Grant applications, sponsor agreements, formal performance reports to funders, funder check-in notes and trackers |
| `district_artifact` | District Artifact | Documents produced by or primarily describing a partner district — district-level data reports, FLDOE materials, district strength profiles |

### Influence & Background

| Type | Display label | Description |
|------|---------------|-------------|
| `field_influence` | Field Influence | Policy briefs, HQIM advocacy materials, external publications authored or co-authored to influence the broader field |

## Why These Tags Matter

The `doc_type` tag does two jobs:

1. **Scopes retrieval** — every vector search and filter in the app and CLI
   narrows by `doc_type` before retrieving chunks. Scoping to
   `feedback_survey_data` vs `qualitative_theming` gives very different
   answers to the same question.

2. **Shapes interpretation** — the agent uses `doc_type` to contextualise
   quotes. A statistic from an `impact_data_report` carries different weight
   than the same number in a `progress_summary`.

---

## How Classification Works (Priority Order)

1. **Filename rules** (`DOC_TYPE_FILENAME_RULES` in `pipeline/get_docs.py`) —
   checked first; keyword patterns in the filename that reliably identify a
   type regardless of which folder the file lives in (e.g. "exit ticket" →
   `feedback_survey_data`, "focus groups report" → `qualitative_theming`).

2. **Folder-path rules** (`DOC_TYPE_RULES` in `pipeline/get_docs.py`) —
   fallback; keywords matched against the second-level folder name
   (e.g. "site visit" folder → `engagement_notes`).

3. **Manual overrides** (`doc_type_overrides.json`) — applied last; file-ID-
   keyed assignments that trump both rules above. Generated by running
   `pipeline/build_overrides.py`.

---

## How to Update Overrides

When a file is classified incorrectly (or a new file lands in the wrong bucket
after ingestion), add an entry to the `exceptions` list in
`pipeline/build_overrides.py`:

```python
# (partial filename fragment,  target doc_type)
("My New Report_2025",  "qualitative_theming"),
```

The fragment is matched case-insensitively against the full filename. It only
needs to be long enough to uniquely identify the file (or group of files that
should all share the same type).

Then regenerate `doc_type_overrides.json`:

```bash
python pipeline/build_overrides.py
```

The script prints matched and unmatched fragments so you can verify coverage.
After regenerating, re-run ingestion or re-ingest only the affected files so
`metadata.json` picks up the new assignments.

> **Note:** Never commit `metadata.json` or `doc_type_overrides.json` changes
> to a shared branch without first running `build_overrides.py` — the JSON
> file is the compiled output of the Python source of truth.
